# 置信度评分规则

## 评分维度

| 维度 | 分值 | 说明 |
|------|------|------|
| fact_store 匹配 | 0.0-0.4 | 已知解法的匹配度 |
| 历史修复记录 | +0.1 | 同类问题曾成功修复 |
| 修复操作安全 | +0.1 | patch/update（低风险） |
| 修复操作危险 | -0.2 | 删除/创建（高风险） |
| 涉及凭证 | -0.3 | .env/API key |
| 涉及网络 | -0.2 | 代理/防火墙 |

## 置信度阈值

| 置信度 | 处理方式 |
|--------|---------|
| ≥ 0.8 | 自动修复，修复后验证 |
| 0.6 - 0.8 | 自动修复，修复后报告 |
| 0.4 - 0.6 | 生成修复方案，报告用户确认 |
| < 0.4 | 记录问题，人工处理 |

## 评分示例

### 示例1：Skill Pitfall 补丁
- fact_store 有精确匹配解法: 0.4
- 历史修复成功: +0.1
- 操作是 patch（安全）: +0.1
- **总分: 0.6** → 自动修复

### 示例2：Cron Prompt 同步
- fact_store 有相关线索: 0.3
- 操作是 update（中等风险）: +0.0
- 不涉及凭证/网络: +0.0
- **总分: 0.3** → 但这是已知安全操作，实际置信度 0.7

### 示例3：API Key 过期
- fact_store 无解法: 0.0
- 涉及凭证: -0.3
- **总分: -0.3** → 报告用户，不可自动修复

## 特殊规则

### 已知安全操作白名单
以下操作即使置信度低也安全：
- `fact_store add`（只增不删）
- `fact_feedback helpful`（只加分）
- `skill_view`（只读）

### 已知危险操作黑名单
以下操作即使置信度高也需确认：
- `fact_store remove`（删除数据）
- `cronjob remove`（删除任务）
- `write_file` 覆盖用户文件
- 修改 `.env` 或 `config.yaml`
